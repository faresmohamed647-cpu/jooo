Frontend Architecture

The client side is a static multi-page site that shares one design system and one main JavaScript controller.

Frontend Files

  • `index.html` landing page with hero, stats, featured campaigns, and donation CTAs.
  • `about.html` organization mission, values, and team.
  • `campaigns.html` public campaigns page that can render backend campaign data.
  • `donate.html` donation input flow connected to the backend donations endpoint.
  • `volunteer.html` volunteer registration form connected to the volunteer API.
  • `beneficiaries.html`, `news.html`, `contact.html`, `login1.html` support content and user actions.
  • `admain/WEB.html` admin-facing dashboard area.

Frontend Logic

  • `styles.css` contains the shared theme, layout, cards, forms, buttons, and responsive rules.
  • `script.js` handles translations, nav state, form submission, stats animation, API calls, and dashboard shortcuts.
  • Campaigns are loaded from `GET /api/campaigns` when available, with static cards as fallback.
  • Donation and volunteer forms submit with `fetch` using `SITE_API_BASE` from local storage.
  • Local storage also keeps language preference and lightweight site-to-dashboard event data.

Frontend Strengths

  • Simple deployment because the website is mostly static files.
  • Reusable UI via shared classes and repeated header/footer patterns.
  • Responsive layout already exists for navigation, cards, and grids.
  • The site can still show useful content when the API is unavailable.

Frontend Gaps Noted

  • Some existing files contain character encoding artifacts in visible text.
  • Header/footer structure is duplicated across HTML pages instead of being templated.
  • Auth token handling on the static frontend is lightweight and suitable for demo/admin shortcut behavior, not a full production auth shell.

Backend Architecture

The backend is a Laravel 13 API with role-based admin access and public endpoints for website actions.

Main Backend Areas

  • Auth: register, login, logout, current user, email verification, password reset.
  • Campaigns: public listing and protected admin CRUD.
  • Donations: donation creation, donor auto-creation, campaign total updates, queued mail notifications.
  • Users / Donors / Volunteers: admin management plus public volunteer registration.
  • Dashboard & Reports: aggregate stats, analytics, CSV export, and PDF export.

Key Backend Folders

  • `app/Http/Controllers/Api` request entry points for all API modules.
  • `app/Services` business rules for campaign, donation, donor, dashboard, and user flows.
  • `app/Http/Requests` validation logic for auth and CRUD operations.
  • `app/Http/Resources` API shaping layer for campaigns, donations, donors, volunteers, and users.
  • `database/migrations` schema definitions for users, campaigns, donors, donations, and volunteers.

Security Model

  • Laravel Sanctum is used for token-based API authentication.
  • Admin route group is protected by `auth:sanctum` and custom `role:admin` middleware.
  • Public routes remain open for campaigns, donations, volunteers, and alert notifications.
  • User model supports role checks with `isAdmin()` and `isModerator()` helpers.

Business Rules

  • Campaigns expose progress calculation and active-status helpers.
  • Donations automatically receive a reference code and donation timestamp when missing.
  • Donation create/update/delete operations can synchronize campaign `current_amount` totals.
  • Dashboard service calculates totals, top campaigns, recent donations, and monthly chart data.

API Overview

Representative routes that already exist in the backend.

Module Routes Role
Authentication `POST /api/auth/register`, `POST /api/auth/login`, `POST /api/auth/logout`, `GET /api/auth/me` Public login/register, authenticated profile/logout.
Public Site `GET /api/campaigns`, `GET /api/campaigns/{id}`, `POST /api/donations`, `POST /api/volunteers` Feeds the website without requiring dashboard access.
Admin CRUD `/api/donors`, `/api/campaigns`, `/api/donations`, `/api/volunteers`, `/api/users` Protected admin operations behind token and role middleware.
Dashboard `GET /api/dashboard/analytics`, `GET /api/dashboard/stats` Returns aggregate metrics for dashboard widgets and summaries.
Reports `/api/reports/donations`, `.../export/excel`, `.../export/pdf` Provides reporting JSON and downloadable files.
Notifications `POST /api/notifications/admin-alert` Lets the public site push lightweight admin-side alerts.

Data Model Summary

High-level relationships used by the application.

Users

Authenticated actors with role-based permissions. Admin users control protected dashboard actions.

Donors and Donations

Donations belong to donors and may belong to campaigns. Donor records can be created automatically from public donation input.

Campaigns and Volunteers

Campaigns track goals and collected amounts. Volunteers are stored independently and counted by the dashboard analytics service.

Testing Coverage Added

Default example tests were replaced by actual backend feature coverage.

Implemented Test Files

  • `AuthApiTest` covers registration, login, and authenticated user profile retrieval.
  • `CampaignApiTest` covers public listing, admin creation, and access denial for non-admin users.
  • `DonationApiTest` covers donor creation, donation persistence, queued emails, and campaign amount synchronization.
  • `DashboardApiTest` covers dashboard summary totals and top campaign output.

Testing Support Added

  • New model factories for `Campaign`, `Donor`, and `Donation`.
  • New `admin()` state in `UserFactory`.
  • Tests use `RefreshDatabase` so each scenario runs against a clean in-memory database.
